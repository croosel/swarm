# Scout-then-Swarm 專案路線圖

> **核心假說**：cross-verification + weighted fusion（交叉驗證 + 加權融合）顯著優於 simple ensemble（簡單集成）。
> **原則**：每一步都有明確的 go/no-go 標準。不要建大教堂，先搭帳篷。

---

## Phase 0: MVP 驗證（一個週末）

**目標**：驗證核心假說 — 交叉驗證 + 加權融合 vs 簡單 ensemble voting

### 範圍

| 做什麼 | 不做什麼 |
|--------|----------|
| 實作 `swarm_judge()` 完整流水線（Scout → Swarm → Verify → Fuse → Learn） | 自由形式任務拆解 |
| 3 個固定模板（code-review, doc-analysis, research）+ 模板匹配 | Pipeline / Checkpoint 執行模式 |
| 2-3 個 Worker 並行執行（Swarm 模式 only） | 任務類型感知評估路由 |
| 交叉驗證（不同模型驗證）+ 第一性原理檢查 | 擺尾舞協議（結構化辯論） |
| 加權融合（0.4 self + 0.6 verify） | 強制探索、經驗 GC |
| SQLite 基本 CRUD + 90 天 TTL | ML 分類器（Fast-Track 用規則引擎） |
| 基礎規則分類器（Fast-Track Gate） | 人類盲評 |
| 100 任務 A/B 測試（swarm_judge vs simple_ensemble） | LangSmith 整合 |

### 成功指標（Go / No-Go）

| 指標 | 目標 | 測量方式 |
|------|------|----------|
| 品質提升 | swarm_judge confidence 比 baseline 高 ≥ 5% | Mann-Whitney U test |
| 統計顯著性 | p < 0.05 | 雙尾檢定 |
| P95 延遲 | < 15 秒 | 100 個任務的 P95 |
| Fast-Track 準確率 | 簡單任務 100% 被分類為 fast_track | 30 個簡單任務命中數 |
| 錯誤率 | < 5% | 100 個任務中失敗數 |
| 平均成本 | < $0.10 USD / 任務 | 總成本 / 100 |

**Go 條件**：以上 6 項全部達成。
**No-Go 條件**：任何 1 項未達成 → 停止或重新設計。

### 產出

- 可執行的 `swarm_judge()` 原型
- `simple_ensemble` 對照組實作
- 100 任務 A/B 基準測試結果
- 統計分析報告與 Go/No-Go 決策

## Phase 1: 任務分解擴展（一個月）

**前提**：Phase 0 Go

**目標**：加入 Orchestrator 與多種執行模式，驗證任務拆解是否進一步提升品質

### 範圍

- [ ] 加入 Pipeline + Checkpoint 執行模式
- [ ] 擴展模板庫到 10+ 個任務類型
- [ ] 任務類型感知評估路由（解決 E1 殘餘風險：分析型/程式碼/創意型用不同評估策略）
- [ ] 經驗 GC + 矛盾檢測
- [ ] 人類盲評 20 任務（swarm vs 單模型最佳）
- [ ] 分解覆核檢查（Decomposition Sanity Check）完整實作

### 成功指標

| 指標 | 目標 | 測量方式 |
|------|------|----------|
| 任務拆解 + 多模型協作 vs 單模型最強 | 提升 ≥ 10% | 人類盲評 1-5 分 |
| 分解覆核通過率 | ≥ 80% | 自動統計 |
| 人類評委偏好 Swarm 結果 | ≥ 60% | 20 任務盲評 |
| P95 延遲（含分解） | < 30 秒 | 統計 |
| 平均成本 | < ¥0.15 / 任務 | 統計 |
| 無任務成本超過 | ¥0.50 | 成本紅線 |

**Go 條件**：人類盲評 ≥ 60% 偏好 Swarm + 成本可控。
**No-Go 條件**：人類盲評 < 50% 或成本失控 → 停止或重新設計。

### 產出

- 人類評估報告
- 擴展模板庫（10+ 任務類型）
- Pipeline + Checkpoint 模式實作
- 經驗 GC 與矛盾檢測機制

## Phase 2: 經驗系統與擺尾舞（三個月）

**前提**：Phase 1 Go

**目標**：加入擺尾舞協議（結構化辯論）與完整經驗系統，驗證持續學習能力

### 範圍

- [ ] 擺尾舞協議（結構化辯論 + 紅隊質詢 + 衰減函數）
- [ ] 強制探索機制（每 10 次執行，強制 1 次不使用經驗庫）
- [ ] 動態模式切換（執行中發現耦合度錯誤時自動調整）
- [ ] 交叉授粉（方向合併）
- [ ] 經驗分級（L1 個人經驗 / L2 統計經驗 / L3 驗證經驗）
- [ ] 模型版本變更時的自動經驗重驗證
- [ ] LangSmith 整合（追蹤 + 可觀測性）

### 成功指標

| 指標 | 目標 | 測量方式 |
|------|------|----------|
| 擺尾舞自然收斂率 | ≥ 70% | 統計（vs 強制收斂） |
| 經驗庫命中率 | ≥ 50% | 統計 |
| 經驗庫命中後品質提升 | ≥ 5% | 對比命中 vs 未命中 |
| 擺尾舞 3 輪內收斂 | ≥ 80% | 統計 |
| stale 經驗占比 | < 30% | 定期驗證 |

**Go 條件**：擺尾舞收斂穩定 + 經驗系統有效提升品質。
**No-Go 條件**：擺尾舞收斂率 < 50% 或經驗系統無效 → 簡化或移除。

### 產出

- 完整 Blueprint 發布
- 擺尾舞協議實作與評估報告
- 經驗系統完整文檔
- LangSmith 整合

## 風險地圖（來自審稿蜂報告）

### 致命風險（1 項）

| ID | 風險 | 應對策略 | 狀態 |
|----|------|----------|------|
| O1 | Orchestrator 任務拆解錯誤 → 全線垃圾進垃圾出 | 模板匹配取代自由拆解 + 分解覆核檢查 + 降級到 Pipeline | Phase 1 解決 |

### 嚴重風險（5 項）

| ID | 風險 | 應對策略 | 狀態 |
|----|------|----------|------|
| E1 | 第一性原理評估對開放式任務無效 | 任務類型感知評估策略（analytical/creative/code） | Phase 1 解決 |
| W1 | 擺尾舞正反饋放大早期錯誤（馬太效應） | 結構化辯論 + 紅隊 + 置信度上限 0.8 + 衰減 + 交叉授粉 | Phase 2 解決 |
| X1 | 經驗過時 / 錯誤經驗擴散 | TTL + model_version 追蹤 + 強制探索 + stale 標記 | Phase 1-2 逐步解決 |
| C1 | 耦合度判斷錯誤 → 產出不一致需重做 | 三級耦合 + 三種執行模式 + 動態模式切換 + 預設保守 | Phase 1 解決 |
| D1 | 延遲 3-5 倍 + 成本 5 倍 | Fast-Track Gate + 延遲預算 + 流式輸出 | Phase 0 已部分解決 |

### 可控風險（2 項）

| ID | 風險 | 應對策略 | 狀態 |
|----|------|----------|------|
| P1 | 與 LangGraph 無本質差異 | 定位為 LangGraph Blueprint；差異化在模板庫 + 辯論協議 + 經驗系統 | 持續關注 |
| M1 | MVP 範圍過大 | 只驗證一個假說（交叉驗證融合 vs 簡單集成），一個週末出結果 | Phase 0 已解決 |
