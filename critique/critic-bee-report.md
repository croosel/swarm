# 審稿蜂報告：Scout-then-Swarm 架構第一性原理挑戰

**審稿人：** Critic Bee
**審稿日期：** 2026-06-17
**審稿對象：** Scout-then-Swarm 多 LLM Agent 協作架構
**審稿立場：** 零信任，純攻擊

---

## 1. 第一性原理評估的可行性

### 攻擊

**核心假設：** 系統能用「邏輯自洽」「回答完整」「有沒有硬傷」作為自動化評估標準。

**Failure Case 1：「邏輯自洽」不等於「正確」。**
一個模型回答「因為 A 所以 B，因為 B 所以 C，因此 A 導致 C」——邏輯完美自洽，但前提 A 本身就是錯的。例如，模型回答「Python 的 GIL 保證執行緒安全，所以多執行緒不需要 lock」，邏輯鏈完整，但結論致命錯誤。自動化評估器如果只看結構，會被「看起來很有道理」的幻覺騙過去。

**Failure Case 2：「回答完整」的標準因任務而異，無法統一。**
- 對「翻譯這段文字」：完整 = 所有句子都翻譯了
- 對「分析這段程式碼的 bug」：完整 = 找到所有 bug？還是主要 bug？誰來定義「足夠完整」？
- 對「幫我寫一個創業計劃」：什麼叫完整？沒有標準答案。

開放式問題根本沒有「完整」的客觀定義。如果評估器說「還不夠完整」，模型可以無限灌水，反而降低品質。

**Failure Case 3：「硬傷」的定義高度依賴領域知識。**
法律文件中的硬傷（引用已廢止的法條）vs 程式碼中的硬傷（SQL injection 漏洞）vs 醫療建議中的硬傷（藥物交互作用）——這三種硬傷的檢測需要完全不同的專業知識。用一個通用的 LLM Judge 去檢測所有領域的硬傷，期望值過高。

**Failure Case 4：第一性原理評估對開放式創造性任務幾乎無效。**
「寫一首詩」「設計一個 logo 方案」「想一個品牌名」——這些任務沒有「邏輯自洽」可言，用第一性原理評估等於拿尺量溫度。

### 嚴重度：**嚴重**

不是致命，因為封閉式任務（程式碼、數學、事實查詢）確實可以用第一性原理評估。但架構宣稱通用性，這裡有嚴重的過度承諾。

### 改進建議

1. **放棄「通用第一性原理評估」的說法，改為「任務類型感知的評估策略」。** 建立一個評估策略路由表：
   - 封閉式任務（程式碼/數學/事實）→ 可用自動化驗證（跑測試、對答案、查來源）
   - 半開放式任務（分析/報告）→ 用 rubric-based 評分（預定義評分維度）
   - 開放式任務（創作/設計）→ 放棄自動評估，只做人類回饋收集
2. **「硬傷檢測」改為模組化外掛：** 程式碼硬傷用 static analysis + 跑測試；法律硬傷用法條資料庫比對；醫療硬傷用藥物交互資料庫。不要指望一個 LLM 能检测所有領域的硬傷。
3. **引入「信心校準」機制：** 評估器不只給分數，還要給信心水平。低信心的評估結果應該標記為「需要人工確認」而非自動決策。

## 2. Orchestrator 的能力瓶頸

### 攻擊

**核心假設：** Orchestrator 能「用經驗 + 第一性原理拆解任務」，並做出正確的分工決策。

**Failure Case 1：任務拆解本身是最難的 AI 任務之一。**
用戶說「幫我寫一個爬蟲，抓取 PTT 所有看板的發文，分析情緒，生成報告」。Orchestrator 需要：
- 判斷這是一個多步任務（爬取 → NLP → 報告）
- 判斷各步驟的依賴關係（爬取和 NLP 是 pipeline，NLP 各看板可以並行）
- 判斷各步驟適合的模型（爬取 → 工具調用能力強的；NLP → 中文理解強的；報告 → 結構化輸出強的）

這需要 Orchestrator 同時具備：任務規劃能力、對每個模型能力的精確理解、對任務耦合度的判斷。**這是 meta-reasoning，比執行任務本身更難。**

**Failure Case 2：拆錯任務 = 全線崩潰。**
如果 Orchestrator 把「分析競爭對手的商業模式」拆成「搜尋對手財報 → 分析財務數據 → 寫報告」，但用戶實際想要的是「分析競爭對手的產品策略」——整個拆解決定了錯誤的方向，後續所有 worker 都在做白工。更糟的是，worker 的輸出看起來很專業（財報分析得很詳細），用戶可能不會意識到方向錯了。

**Failure Case 3：Orchestrator 模型選擇的兩難。**
- 用 Qwen 3.7 Max 當 Orchestrator（最強智能體模型）→ 每次任務先花一筆高額費用在拆解上，簡單任務得不償失
- 用 MiniMax M3 當 Orchestrator（便宜）→ 拆解品質不穩，複雜任務容易拆錯
- 用規則引擎（不用 LLM）→ 只能處理預定義的任務類型，遇到新任務類型就癱瘓

**Failure Case 4：Orchestrator 成為單點故障。**
整個架構的品質上限被 Orchestrator 鎖死。如果 Orchestrator 的拆解準確率是 80%，那整條流水線的品質上限就是 80%——不管 worker 多強。

### 嚴重度：**致命**

Orchestrator 是整個架構最脆弱的環節。它的任務（meta-reasoning + task decomposition）是目前 LLM 最不擅長的任務之一。拆錯的後果是垃圾進垃圾出，而且錯誤難以被下游發現。

### 改進建議

1. **不要讓 Orchestrator 做自由形式的任務拆解。改為「模式匹配 + 模板」：**
   - 預定義 20-30 種常見任務模板（程式碼生成、深度研究、文件分析、資料處理...）
   - Orchestrator 的工作簡化為：分類（這是哪類任務？）+ 填模板（用哪個模板？）
   - 分類任務比自由拆解容易一個數量級
2. **引入「拆解確認」機制：** Orchestrator 拆解完成後，把拆解方案展示給用戶確認，而非直接執行。至少在 MVP 階段，這一步不可省略。
3. **Orchestrator 用「強模型 + 結構化輸出」：** 強制要求 Orchestrator 用 JSON Schema 輸出拆解方案，包含：子任務列表、依賴關係圖、各子任務的預期輸出格式。結構化輸出可以大幅降低自由文本拆解的幻覺風險。
4. **建立「拆解回饋迴路」：** 每次任務結束後，記錄拆解方案 vs 實際執行結果的差異，作為 Orchestrator 的 fine-tuning 資料。

## 3.「搖擺舞」正回饋的風險

### 攻擊

**核心假設：** 借鑒蜜蜂的搖擺舞機制，用正反饋放大優質方案。

**Failure Case 1：馬太效應——早期錯誤被放大。**
三個 worker 模型回答同一個問題。MiniMax M3 第一個返回答案，品質 60 分。搖擺舞機制給它初始權重。Qwen 3.7 Max 第二個返回，品質 80 分，但因為 M3 已經累積了正反饋，Qwen 的權重被壓低。最終融合時 M3 的低品質答案佔了更大比重。

蜜蜂世界裡，第一隻找到花蜜的蜜蜂確實可能找到的是最好的——因為花蜜是客觀存在的。但 LLM 世界裡，**先返回的答案不代表更好，只代表模型推理更快**。速度與品質的相關性很弱。

**Failure Case 2：贏者通吃壓制少數派正確意見。**
假設三個模型分析一段程式碼：
- M3 和 K2.7 都認為「這段程式碼沒有 bug」（多數派）
- Qwen 3.7 Max 認為「有一個 race condition」（少數派）

搖擺舞的正回饋會讓多數派意見被放大，少數派意見被壓制。但在程式碼審查中，少數派的警報往往是正確的——bug 通常就是大多數人忽略的那個。

**Failure Case 3：衰減參數的調校是黑魔法。**
搖擺舞需要衰減機制防止正反饋失控。但：
- 衰減太快 → 信號被抹平，等於沒有正反饋，回到平均主義
- 衰減太慢 → 早期優勢被過度放大，回到馬太效應
- 衰減速度本身應該因任務類型而異（事實查詢需要快速收斂，開放式分析需要更長時間的「辯論」）

**沒有一組固定參數能同時適用於所有場景。**

**Failure Case 4：類比失效——蜜蜂和 LLM 的決策環境根本不同。**
蜜蜂是在物理世界中做選擇（這朵花有蜜 vs 那朵花有蜜），選項數量有限，評估標準單一（蜜的品質）。LLM 協作是在無限維度的答案空間中做融合，選項不是離散的，評估標準是多維且模糊的。**把生物類比直接搬到 AI 系統，是一種概念上的偷懶。**

### 嚴重度：**嚴重**

正回饋機制在錯誤方向上的放大效應，可能導致系統性的偏差。特別是在安全關鍵場景（醫療、法律），壓制少數派正確意見的後果很嚴重。

### 改進建議

1. **取消純正反饋機制，改為「結構化辯論」：**
   - 每個 worker 不只給答案，還要給出對自己答案的信心分數和關鍵假設
   - 引入一個「Devil's Advocate」角色，專門挑戰多數派意見
   - 最終決策不是誰的權重大，而是誰的論證更經得起挑戰
2. **少數派保護機制：**
   - 如果某 worker 的意見與多數派顯著不同，不要直接壓低，而是觸發「深入驗證」：讓該 worker 展開論證，同時讓其他 worker 嘗試反駁
   - 類似法庭制度：少數派有權獲得充分的聽證時間
3. **放棄「搖擺舞」這個類比，改用「預測市場」模型：**
   - 每個 worker 對自己答案的正確性「下注」
   - 如果後續驗證證明某個 worker 經常是正確的少數派，它的「信譽分」會提高
   - 這比純正反饋更robust，因為它基於歷史準確率而非當前輪次的信號強度
4. **如果一定要保留正回饋，至少加入「多樣性閾值」：**
   - 如果三個 worker 的答案高度一致（餘弦相似度 > 0.9），正反饋可以放大
   - 如果答案分歧大，禁止正反饋，改為強制辯論

## 4. 經驗回饋的陷阱

### 攻擊

**核心假設：** LLM Wiki 累積經驗，讓任務拆解越來越精準。

**Failure Case 1：經驗過時——模型更新後舊經驗可能有害。**
Wiki 記錄了「Kimi K2.6 處理中文長文本效果最好，優先分配中文任務」。但系統已升級到 K2.7，且 K2.7 的中文能力可能有不同的特性（也許更擅長程式碼而非散文）。舊經驗導致錯誤的路由決策。

更具體地說：
- 模型提供商經常靜默更新模型（API 端點不變，但底層模型已換）
- 模型在不同任務上的表現會隨版本更新而變化
- 舊經驗的半衰期可能只有幾週到幾個月

**Failure Case 2：錯誤經驗的寫入和擴散。**
任務執行結果被記錄為「成功」並寫入 Wiki。但如果：
- 任務評估本身有誤（評估器給了高分但答案實際有問題）
- 任務是邊緣案例但被記錄為通用經驗
- 執行環境有特殊性（恰好用了某個模型的最佳版本）

錯誤經驗一旦寫入 Wiki，會被後續的 Orchestrator 反覆引用，形成**錯誤經驗的自我強化迴路**。

**Failure Case 3：經驗累積導致保守化——創新停滯。**
Wiki 記錄了「任務類型 X 用模板 A 效果最好」。Orchestrator 每次都選模板 A。但實際上，隨著模型能力進化，模板 B（之前沒試過的新方案）可能已經更好了。

這就是 recommendation system 中經典的 **exploration-exploitation dilemma**。純粹基於歷史經驗的系統，會陷入局部最優解。

**Failure Case 4：Wiki 的維護成本被低估。**
- 誰來清理過時經驗？
- 誰來合併衝突經驗？（「中文任務給 K2.7」vs「中文任務給 Qwen 3.7 Max」——兩條經驗可能在不同時期都是對的）
- 誰來評估經驗的泛化性？（一次成功的經驗不代表通用規律）

如果沒有人維護，Wiki 會變成一個充滿過時和矛盾資訊的垃圾場。

### 嚴重度：**嚴重**

不是致命，因為經驗系統確實有價值。但如果設計不當，它會從資產變成負債。目前架構中沒有提到任何經驗驗證、清理和更新機制，這是一個重大遗漏。

### 改進建議

1. **經驗條目必須帶有元數據：**
   - 建立時間
   - 使用的模型版本
   - 樣本量（基於多少次執行得出的結論）
   - 信心分數
   - 有效期（超過 N 天自動標記為「待驗證」）
2. **引入「經驗挑戰」機制：**
   - 定期（例如每週）用當前模型重新執行歷史經驗對應的任務
   - 如果結果與經驗不符，自動標記為「過時」
   - 這類似 A/B testing 的持續驗證
3. **強制 exploration：**
   - 設定 10-20% 的任務不使用 Wiki 經驗，而是隨機嘗試不同的拆解策略
   - 如果新策略效果更好，寫入 Wiki 作為新經驗
   - 這解決了保守化問題
4. **經驗分級：**
   - L1（個人經驗）：基於 1-5 次執行，低信心，僅供參考
   - L2（統計經驗）：基於 20+ 次執行，中信心，可用於路由
   - L3（驗證經驗）：經過人工確認，高信心，可用於生產決策
   - 不同等級的經驗有不同的權重和過期策略

## 5. 弱耦合 vs 強耦合的判斷

### 攻擊

**核心假設：** 系統能自動判斷任務的耦合度，弱耦合走 Swarm（並行），強耦合走 Pipeline（順序）。

**Failure Case 1：耦合度判斷本身就是一個 hard problem。**
用戶說「比較 React 和 Vue 的優缺點，用兩個框架分別實現同一個功能，然後寫一個對比報告」。

這是弱耦合還是強耦合？
- 「分別實現」→ 看起來弱耦合，可以並行
- 「對比報告」→ 需要知道兩邊的實現細節才能寫，實際上是強耦合
- 更深層：如果 React 實現中發現了一個架構問題，可能需要調整 Vue 的實現方式來做公平比較

**耦合度不是二元的，而是動態變化的。** 任務執行過程中可能發現新的依賴關係。

**Failure Case 2：判斷錯誤的代價不對稱。**
- 弱耦合誤判為強耦合（用了 Pipeline）→ 只是慢一些，結果正確
- 強耦合誤判為弱耦合（用了 Swarm）→ worker 並行執行，但互相不知道對方的決策，產出不一致。例如：
  - Worker A 設計了資料模型的 schema
  - Worker B 同時在寫依賴該 schema 的 API
  - 兩個 worker 用了不同的命名規範和資料類型
  - 最終整合時發現根本對不上，需要重做

**第二種錯誤的代價是重做，遠大於第一種錯誤的代價（只是多等幾秒）。**

**Failure Case 3：忽略了「中度耦合」這個多數情況。**
現實中大部分任務不是純弱耦合也不是純強耦合。例如「寫一個帶資料庫的 web app」：
- 前端和後端可以並行（弱耦合）
- 但 API 契約需要先定義（強耦合的前置條件）
- 資料庫 schema 影響前後端（共享依賴）

這種「部分並行 + 部分順序 + 共享上下文」的任務，用 Swarm 或 Pipeline 都不完全合適。

**Failure Case 4：耦合度可能因模型能力而異。**
- 如果 worker 模型有很強的上下文理解能力，給它足夠的背景資訊就能獨立工作 → 原本強耦合的任務變成弱耦合
- 如果 worker 模型容易偏離方向，需要頻繁對齊 → 原本弱耦合的任務需要變成強耦合

架構沒有考慮「模型能力」對耦合度判斷的影響。

### 嚴重度：**嚴重**

耦合度判斷錯誤（特別是強耦合誤判為弱耦合）會導致產出不一致和重做。這在實際使用中可能非常頻繁。

### 改進建議

1. **放棄二元判斷，引入三種執行模式：**
   - **Swarm（純並行）：** 子任務完全獨立，不需要共享上下文
   - **Pipeline（純順序）：** 子任務有明確的前後依賴
   - **Checkpoint（混合）：** 先並行執行，到指定檢查點會合對齊，再繼續並行
   - 大部分任務應該走 Checkpoint 模式
2. **預設從保守開始：**
   - 不確定耦合度時，預設用 Pipeline（寧可慢，不要錯）
   - 只有在有充分證據（來自 Wiki 經驗）表明可以並行時，才用 Swarm
   - 這利用了前面提到的「判斷錯誤代價不對稱」——保守錯誤的代價低
3. **引入「動態耦合」機制：**
   - 任務開始時預設為 Pipeline
   - 執行過程中如果發現子任務之間沒有實際依賴，動態切換為並行
   - 這比靜態判斷更靈活
4. **API 契約先行：**
   - 對於中度耦合任務，第一步永遠是定義子任務之間的「契約」（輸入輸出格式、命名規範）
   - 契約定義好之後，子任務就可以安全地並行執行
   - 這解決了大部分的中度耦合問題

## 6. 延遲和成本

### 攻擊

**核心假設：** 四階段（Scout + Swarm + Verify + Learn）的開銷是可接受的。

**具體延遲分析（以一個中等複雜度的任務為例）：**

| 階段 | 操作 | 預估延遲 |
|------|------|---------|
| Scout | Wiki 搜尋 + Orchestrator 拆解 | 3-5 秒 |
| Swarm | 3 個 worker 並行生成 | 10-20 秒（取最慢的） |
| Verify | Judge 交叉驗證 + 第一性原理評估 | 5-10 秒 |
| Learn | 結果寫回 Wiki | 2-3 秒 |
| **總計** | | **20-38 秒** |

**對比：直接調用單模型 = 3-8 秒。**

**Scout-then-Swarm 比單模型慢 3-5 倍。**

**Failure Case 1：用戶體驗災難。**
用戶在 IDE 中問一個程式碼問題，等 30 秒才得到回答。VSCode Copilot 等 2 秒就給出建議。即使你的答案品質高 10%，用戶也不會等。

**延遲是產品殺手。** 在實際使用中，用戶對延遲的容忍度遠低於對品質的追求。

**Failure Case 2：簡單任務的過度工程。**
用戶問「Python 的 list comprehension 語法是什麼？」——這種任務走完整四階段：
- Scout：Wiki 搜尋 → 找到「Python 語法查詢」模板
- Swarm：三個模型分別回答同一個基礎問題
- Verify：Judge 驗證三個基本一致的答案
- Learn：記錄「Python 語法查詢用單模型即可」

整個過程花了 20 秒，浪費了 3 次多餘的 API 調用，只為了得到一個 Google 搜尋就能回答的答案。

**Failure Case 3：成本在規模化時失控。**
假設日均 1000 次任務調用：
- 每次任務平均 5 次 API 調用（Orchestrator + 3 Worker + Judge）
- 日均 5000 次 API 調用
- 月均 150,000 次
- 如果平均每次調用成本 ¥0.0002（混合模型），月成本 = ¥30

看起來不貴？但這是假設所有任務都中等複雜度。如果用戶用這個系統做深度研究（每個任務可能需要 10+ 次子調用），月成本輕鬆翻 5-10 倍。

**而且這個成本還沒算 Orchestrator 的開發和維護成本、Wiki 系統的基礎設施成本、監控系統的运营成本。**

**Failure Case 4：延遲和成本的 ROI 不清楚。**
花 3-5 倍的延遲和 5 倍的成本，換來多少品質提升？如果沒有 benchmark 數據證明品質提升 > 30%，這個 ROI 是負的。

### 嚴重度：**嚴重**

延遲和成本問題不會讓架構「不可行」，但會嚴重限制適用場景。如果不解決，系統只能用在「不在乎延遲且任務足夠複雜」的小眾場景。

### 改進建議

1. **引入「快速通道」——大部分任務不該走完整四階段：**
   - 簡單任務（分類器判斷）→ 直接單模型回答，跳過 Scout/Swarm/Verify/Learn
   - 中等任務 → Scout + 單模型 + 輕量驗證（不做完整 Swarm）
   - 複雜任務 → 完整四階段
   - **80% 的用戶任務應該走快速通道**
2. **延遲預算制：**
   - 用戶或系統設定延遲上限（如 5 秒）
   - 系統在預算內盡力而為，超時直接返回當前最佳結果
   - 這比固定四階段更實際
3. **快取層：**
   - 相似任務的結果快取，直接返回
   - Wiki 搜尋結果快取
   - 模型能力的路由決策快取（不需要每次都重新分析）
4. **成本透明化：**
   - 每次任務顯示實際成本和延遲
   - 讓用戶決定「這個任務值不值得花這麼多錢/時間」
   - 用戶可以選擇「省錢模式」（單模型）vs「高品質模式」（完整 Swarm）

## 7. 與現有方案的競爭定位

### 攻擊

**核心假設：** Scout-then-Swarm 與 LangGraph、CrewAI、AutoGen 等框架有本質區別。

**殘酷現實：**

**LangGraph 已經能做：**
- 任意 DAG 任務圖（包含 Swarm 和 Pipeline）
- 狀態管理、條件分支、迴圈
- 人類介入節點
- 持久化、時間旅行調試

**CrewAI 已經能做：**
- Agent 角色定義
- 任務委派
- 順序/並行執行
- 工具使用

**AutoGen 已經能做：**
- 多 Agent 對話
- 群聊模式
- 自動回覆和人類介入
- 程式碼執行

**Scout-then-Swarm 所謂的「獨特價值」是什麼？**
1. 「蜜蜂搖擺舞」→ 本質就是一個加權投票/正反饋機制，LangGraph 幾行程式碼就能實現
2. 「第一性原理評估」→ 本質就是 LLM-as-Judge，所有框架都支援
3. 「LLM Wiki 經驗回饋」→ 本質就是 RAG + 記憶系統，LangChain 早就做了
4. 「弱耦合/強耦合自動判斷」→ 本質就是任務路由，CrewAI 也有

**Failure Case 1：換了個名字的同一件事。**
如果把「搖擺舞」改成「加權正反饋投票」，把「Scout」改成「任務分析器」，把「Wiki」改成「RAG 記憶」，把「法定人數閾值」改成「信心閾值」——這個架構就是 LangGraph + RAG 的一個 preset。沒有新東西。

**Failure Case 2：沒有 benchmark 就沒有說服力。**
- DRACO benchmark 是 OpenRouter 用來評測 Fusion 的，不是用來評測「多 Agent 協作架構」的
- 沒有 SWE-bench、HumanEval、MMLU 等標準 benchmark 上的數據
- 沒有與 LangGraph 相同任務的頭對頭比較
- 沒有 A/B 測試數據

**「理論上更好」不足以說服任何人遷移。** 現有框架已經有成熟的生態、文檔、社群、整合工具。

**Failure Case 3：重新造輪子的機會成本。**
開發 Scout-then-Swarm 需要的工程量（估計 2-3 人月），如果投入在 LangGraph 上做定制開發（寫一個好的 task decomposition node + memory integration），可能早就上線了。

### 嚴重度：**嚴重**

不是致命——如果做得好，它可以是一個 LangGraph 上的應用層方案。但如果定位為「全新架構」，會面臨「為什麼不用 LangGraph」的質問而無法回答。

### 改進建議

1. **不要做框架，做 LangGraph 上的一個 Blueprint：**
   - 基於 LangGraph 實現，把 Scout-then-Swarm 包裝成一個可複用的 Graph 模板
   - 這樣可以借力 LangGraph 的基礎設施（狀態管理、調試、監控），專注在差異化邏輯上
   - 用戶遷移成本為零——他們已經在用 LangGraph 了
2. **明確差異化定位：**
   - 不要說「我是新的多 Agent 框架」
   - 說「我是第一個把蜂群決策理論應用到 LLM Agent 編排的 Blueprint」
   - 重點是「決策品質」，不是「Agent 協作」
3. **第一天就要有 benchmark：**
   - 選 3-5 個標準 benchmark（SWE-bench for code, MMLU for reasoning, TruthfulQA for hallucination）
   - 跑頭對頭比較：單模型 vs 簡單 ensemble vs Scout-then-Swarm
   - 如果沒有顯著提升，重新思考架構
4. **擁抱現有生態，不要對抗：**
   - 路由邏輯 → LangGraph 的 conditional edges
   - 記憶系統 → LangGraph 的 checkpointing + 外部 vector store
   - Agent 通訊 → AutoGen 的 group chat 或 LangGraph 的 multi-agent 模式
   - 你的獨有价值 = 搖擺舞決策算法 + 經驗學習閉環，把力量集中在這兩點

## 8. MVP 的可行性

### 攻擊

**核心假設：** 可以做出一個 MVP 驗證這個架構。

**現實估算：**

完整架構需要的組件：
1. Orchestrator（任務拆解引擎）
2. Worker Pool（多模型調用 + 負載均衡）
3. 搖擺舞決策引擎（正反饋 + 衰減 + 法定人數）
4. 第一性原理評估器
5. Wiki 經驗系統（RAG + 寫回 + 驗證）
6. 耦合度分類器
7. Pipeline/Swarm 執行引擎
8. 監控和調試工具
9. 配置管理系統
10. 錯誤恢復和重試機制

**這不是 MVP，這是一個產品。** 做完這些需要至少 3-6 人月。

**Failure Case 1：MVP 範圍太大，永遠做不完。**
如果「MVP」包含以上所有組件的簡化版，結果是每個組件都不成熟，整體體驗差。與其做 10 個 60 分的組件，不如做 2 個 90 分的組件。

**Failure Case 2：核心假設沒有被優先驗證。**
整個架構有兩個核心假設：
- A：「蜂群決策（搖擺舞 + 交叉驗證）比簡單 ensemble 品質更好」
- B：「經驗回饋讓任務拆解越來越精準」

如果 A 不成立，整個架構沒有存在價值。如果 B 不成立，系統只是一個靜態的 task router。

但大部分開發時間會花在基礎設施（模型調用、狀態管理、錯誤處理）上，而不是驗證 A 和 B。

**Failure Case 3：沒有定義「什麼結果算成功」。**
MVP 做完了，怎麼判斷成功？
- 品質比單模型好 10%？20%？
- 延遲在多少秒以內可接受？
- 用戶願意為品質提升付出多少延遲代價？

沒有預定義的成功標準，MVP 的結果會被主觀解讀，無法做出 go/no-go 決策。

### 嚴重度：**可控**

MVP 本身不難做，難的是做對。需要極度精簡範圍。

### 改進建議

1. **MVP 只驗證一個假設：「交叉驗證 + 加權融合 vs 簡單 ensemble voting」**

   具體做法：
   ```
   輸入 → 3 個模型並行回答 → 簡單平均融合（baseline）
   輸入 → 3 個模型並行回答 → Judge 交叉驗證 + 加權融合（experiment）
   ```
   跑 100 個標準任務，比較兩組結果。**這個 MVP 一個週末就能做完。**

2. **如果第一步驗證成功，第二步加入「任務拆解」：**
   ```
   輸入 → Orchestrator 拆解 → 分派給不同模型 → 整合結果
   輸入 → 直接給最強模型
   ```
   比較拆解後多模型協作 vs 單模型直接回答。

3. **經驗系統是最後才加的：**
   - 前兩步驗證成功後，手動記錄哪些任務模板效果好
   - 用這些手動經驗做一個靜態路由表
   - 如果靜態路由表有效，再考慮自動化（Wiki）

4. **MVP 成功標準（預定義）：**
   - 在 100 個測試任務上，交叉驗證 + 加權融合 比 簡單 ensemble 提升 ≥ 5%
   - 任務拆解 + 多模型協作 比 單模型最強 提升 ≥ 10%
   - 端到端延遲 < 15 秒（P95）
   - 如果達不到，放棄或重新設計

5. **MVP 技術棧：**
   - 不要自己造輪子
   - 用 LangGraph 做任務編排
   - 用 LiteLLM 做模型調用統一接口
   - 用 SQLite 做經驗存儲（MVP 不需要向量資料庫）
   - 一個 Python 腳本 + 一個 Jupyter notebook 就夠了

## 最終建議：如果只能做 20%，應該做什麼

**答案：只做「Judge 交叉驗證 + 加權融合」這一個模塊。**

原因：
1. 這是整個架構中**最有可能是真正創新**的部分
2. 它是**最容易驗證**的——100 個任務，A/B 比較，一個週末出結果
3. 如果驗證成功，它**可以獨立使用**——不需要 Scout、Wiki、耦合度判斷等基礎設施，直接嵌入任何現有的 LLM 調用流程
4. 如果驗證失敗，**整個架構就沒有存在價值**——早發現早止損

**具體 MVP 方案：**

```python
# 偽程式碼：最小可行驗證
def swarm_judge(query, models, judge_model):
    # Step 1: 並行調用
    answers = parallel_call(models, query)

    # Step 2: Judge 交叉驗證
    evaluation = judge_model.evaluate(
        query=query,
        answers=answers,
        criteria=["factual_accuracy", "completeness", "logical_consistency"]
    )

    # Step 3: 加權融合
    final = judge_model.synthesize(
        query=query,
        answers=answers,
        weights=evaluation.weights,
        flags=evaluation.red_flags  # 標記的問題點
    )

    return final

# 驗證：跑 100 個任務
# Baseline: simple_ensemble(query, models) — 簡單多數投票
# Experiment: swarm_judge(query, models, judge_model) — 交叉驗證融合
# 指標: 人工評分 1-5, 兩組 blind 比較
```

如果 swarm_judge 在 100 個任務上比 simple_ensemble 顯著更好（p < 0.05），繼續投入。否則，停止。

## 總結

Scout-then-Swarm 架構的核心問題不是「想法不好」，而是**過度設計 + 缺乏驗證**。

它試圖一次性解決五個問題（任務拆解、模型協作、決策融合、經驗學習、耦合判斷），每個都是一個獨立的難題。把五個難題捆在一起做，失敗概率是指數級的。

**正確的順序是：**

1. 先驗證「交叉驗證融合」是否比「簡單 ensemble」好 → 一個週末
2. 如果好，加入「任務拆解」看是否進一步提升 → 一個月
3. 如果好，加入「經驗系統」看是否隨時間改善 → 三個月
4. 每一步都有明確的 go/no-go 標準

**不要建大教堂。先搭帳篷，看看有沒有人想住進來。**
